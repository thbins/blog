| **인프라 운영** | **장점** | **단점** |
| --- | --- | --- |
| 매뉴얼 | 물리적인 특성으로 즉시 확인 및 즉시 전달 가능 | 변경 사항 반영이 어렵고 실행을 위한 별도 요소 필요 |
| 스크립트 | 반복 작업을 줄이고 사용이 간단 | 순차적으로만 실행되고 시간이 지나면서 최종 상태가 일치하지 않음 |
| 가상 머신 | 미리 구성된 템플릿을 사용하여 관리와 확장이 용이 | 가상화 범위 외적인 요소는 자동화가 불가능하고 하이퍼바이저에 의존적 |
| 클라우드 | API를 통해 더 많은 인프라를 가상화하고 자동화 도구 제공 | 클라우드 제공자마다 서로 상이한 API가 제공되어 각각을 위한 개별적 작업 필요 |
| 컨테이너 | 상호 호환되는 컨테이너 런타임을 통해 서비스 배포 자동화 및 높은 이동성 | 애플리케이션 배포 자동화를 위한 추가 플랫폼 레이어의 관리 추가 |

위 표는 인프라의 구성 방식과 변경 내용을 공유하고 자동화하기 위한 단계별 요소들입니다.

<br/>

자동화를 위한 노력은 처음에는 단순히 반복되는 작업을 줄이려는 **단일 프로세스 중심**이었습니다. 하지만 시간이 흐르면서 점차 **여러 팀과 조직이 함께 협업하는 엔지니어링 기반**으로 확장되었습니다.

자동화에서 중요한 건 각 **프로세스**를 단순히 묶는 것이 아니라, 이를 **재활용할 수 있도록 통합**하는 일입니다. 프로세스를 **워크플로(Workflow)** 형태로 설계하면 전체 동작이 어떻게 흘러가는지 쉽게 파악할 수 있고, 필요할 때 더 나은 방식으로 바꾸는 것도 훨씬 수월해집니다.

과거의 자동화는 주로 **기술 주도형 접근**에 의존했습니다. 엔지니어들이 협력해 빅뱅 방식(한 번에 크게 적용하는 방식)으로 자동화를 추진했지만, 절차는 느리고 복잡했으며 특정 환경에 맞춰진 **맞춤형 자동화**가 많았습니다. 이 때문에 내부 인프라 관리에는 적합했지만, 범용성은 부족했습니다.

그러나 오늘날은 상황이 달라졌습니다. **클라우드 환경**처럼 계속 변하고, 여러 플랫폼이 융합되는 환경에서는 기존 자동화를 그대로 재활용하기 어렵습니다. 따라서 이제는 **프로세스 중심 접근법**이 필요합니다. 작은 단위로 독립적으로 동작하는 **마이크로서비스 아키텍처(MSA)**와 **분산 시스템** 같은 구조를 지향하면서, 빠른 시장 적응력과 장애 대응 능력을 확보해야 합니다. 또한 개별 서비스를 릴리스하는 과정에서 반복되는 일을 줄이기 위해서는 사람이 직접 조작하는 대신 **코드 기반 자동화**를 설계하는 것이 핵심이 됩니다.

여기서 등장하는 개념이 바로 코드형 인프라(IaC, Infrastructure as Code)입니다. 인프라를 코드로 표현하고, 그 코드가 곧 인프라의 정의서가 되는 방식입니다. 이제는 UI(그래픽 화면)나 CLI(명령어)로 수동 작업을 하는 대신, 코드로 인프라를 관리하는 것이 일반적인 접근이 되었습니다.

다만, 자동화와 협업 기반 기술의 발전이 무조건 단순화를 보장하지는 않습니다. 소프트웨어 기술의 성장 덕분에 자동화가 가능해졌지만, 기존의 **하드웨어 기반 운영**에 비해 오히려 더 많은 시간과 인력이 소모되기도 합니다. 그래서 이를 효율적으로 관리할 수 있도록 **API 같은 인터페이스**가 제공됩니다.

이론적으로는 자동화와 가상화, 컨테이너화가 업무량을 줄이고 더 나은 워라밸을 보장해야 합니다. 그러나 실제로는 자동화 과정에서 수많은 반복과 검증이 필요해, 이전보다 더 많은 시간과 노력이 요구되기도 합니다.

또한 주목할 점은, 자동화가 발전했음에도 불구하고 **기존 인프라 관리 방식은 여전히 모두 공존**하고 있다는 것입니다. 퍼블릭과 프라이빗 클라우드가 도입되었지만 동시에 가상 머신(VM)과 베어메탈 서버도 여전히 쓰이고 있습니다. 게다가 기존 서버보다 더 작은 단위의 **엣지 컴퓨팅**과 **IoT 인프라**까지 관리 대상에 포함되고 있습니다.

즉, 자동화는 계속 진화하고 있지만, 그 과정에서 관리 대상은 오히려 늘어나고 있고, 새로운 접근과 더 유연한 설계가 반드시 필요해지고 있습니다.

<br/>

**용어 설명**

-   **프로세스(Process) :** 특정 목표를 위해 정해진 단계에 따라 진행되는 작업의 흐름.
-   **워크플로(Workflow) :** 여러 프로세스를 연결해 전체적인 흐름을 시각화하거나 설계한 것. 업무 자동화 설계도 같은 개념.
-   **빅뱅 방식(Big Bang Approach) :** 작은 단위로 나눠 점진적으로 적용하는 게 아니라, 한 번에 크게 전체 시스템에 적용하는 방식.
-   **클라우드(Cloud) :** 인터넷 기반으로 서버, 스토리지, 네트워크 등을 필요할 때 빌려 쓰는 IT 인프라 환경.
-   **분산 시스템(Distributed System) :** 여러 대의 컴퓨터가 네트워크로 연결되어 하나의 시스템처럼 동작하는 구조.
-   **마이크로서비스 아키텍처(MSA, Microservices Architecture) :** 서비스를 작은 단위로 쪼개어 각각 독립적으로 개발, 배포, 운영할 수 있도록 만든 소프트웨어 구조.
-   **코드형 인프라(IaC, Infrastructure as Code) :** 서버, 네트워크, 스토리지 같은 인프라 자원을 코드로 정의하고 관리하는 방식. 대표적으로 Terraform, AWS CloudFormation 같은 도구가 있음.
-   **API (Application Programming Interface)**: 소프트웨어끼리 상호작용할 수 있도록 해주는 인터페이스.
-   **가상화(Virtualization)**: 물리 서버 위에 여러 가상 서버(VM)를 올려 자원을 효율적으로 활용하는 기술.
-   **컨테이너(Container)**: 애플리케이션과 실행 환경을 패키징해 어디서든 동일하게 실행할 수 있도록 한 단위.
-   **베어메탈 서버(Bare Metal Server)**: 가상화 없이 물리 서버를 직접 사용하는 환경.
-   **엣지 컴퓨팅(Edge Computing)**: 데이터 처리를 중앙 서버가 아닌 사용자 가까운 장치나 네트워크 엣지에서 수행하는 방식.
-   **IoT (Internet of Things)**: 센서와 장치가 인터넷에 연결되어 데이터를 주고받는 사물인터넷 기술.
